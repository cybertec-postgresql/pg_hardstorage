# pg_hardstorage v1.0

> The first stable release.  Every advertised v1.0 feature
> is implemented, every quality gate is wired, and the
> documentation site that goes with it lands at the same
> time.

`pg_hardstorage` v1.0 is the result of ~30 development
batches and 27 audit cycles since v0.9.  The headline:
**you can pick any of the three major clouds + on-prem
HSM** and get a code-change-free deployment, with a backup
data plane that talks the PostgreSQL replication protocol
(so it needs no SSH or OS-level access to the database host —
fully-managed DBaaS that withhold `BASE_BACKUP`, such as RDS
or Cloud SQL, are out of scope).

The v1.0 commitment: **24-month backward compatibility on
every on-disk and on-the-wire schema** — manifests, config,
output JSON, on-disk chunk envelope.  Operators who write
scripts against v1.0 keep working through 2028.

Apache 2.0.  PostgreSQL 15+, with PG 17 as the default
sandbox version and PG 18 supported with allow-failure CI
guardrails.

## Headline features

### 1. Five Tier-1 KMS providers

The KMS surface that started v0.9 with AWS KMS now covers
the cloud providers that matter plus on-prem HSM:

  - **AWS KMS** (`aws-kms://`) — IAM, FIPS endpoints, key
    aliases.
  - **GCP KMS** (`gcp-kms://`) — ADC chain, key-ring +
    key-name URL form.
  - **Azure Key Vault** (`azure-kv://`) — `azidentity`
    chain, sovereign clouds via dotted vault host.
  - **HashiCorp Vault Transit** (`vault-transit://`) —
    AppRole + token + namespace; deletion gated by
    `deletion_allowed`.
  - **PKCS#11 / HSM** (`pkcs11://`) — covers Thales
    nCipher, Utimaco, AWS CloudHSM, YubiHSM2, SoftHSM2;
    AES-GCM (default) and RSA-OAEP wrap mechanisms.  Built
    behind the `pkcs11` build tag (CGO + `libpkcs11`).

KEK material never leaves the cloud / device; every
WrapDEK / UnwrapDEK round-trips through it.  The
[KEKRef schemes reference](../reference/kekref-schemes.md)
documents per-provider URL forms, auth chains, FIPS
posture, and Shred semantics.

### 2. Five Tier-1 storage backends

Three major hyperscalers + on-prem fs + SFTP.  Every
backend supports the same `StoragePlugin` contract: native
ConditionalPut, native rename, WORM mode where the backend
allows it.

  - **fs** (`file://`) — local filesystem; the default.
  - **s3** (`s3://`) — IAM-aware, S3 Express One Zone +
    standard, server-side encryption supported.
  - **gcs** (`gcs://`) — ADC chain, native
    `If: Conditions{DoesNotExist: true}`.
  - **azblob** (`azblob://`) — `If-None-Match: *`
    precondition, sovereign clouds, access-tier hot/cold/
    cool/archive.
  - **sftp** (`sftp://`) — POSIX rename via the
    `posix-rename@openssh.com` extension; key-based auth
    with the standard SSH ladder.

The [storage-URL-schemes reference](../reference/storage-url-schemes.md)
covers the URL form, query params, auth chain, and
capabilities for each.

### 3. Native WAL streaming + Patroni-aware failover

The data plane is the **PostgreSQL replication protocol**
over a database connection — no `archive_command` shim
required.  The agent issues `START_REPLICATION` against a
persistent physical slot and the slot's
`confirmed_flush_lsn` only advances after a chunk batch
commits, so a crash between commits replays safely on
restart.

Patroni clusters get four cooperating mechanisms:

  - **REST awareness** — the agent polls
    `patroni-leader:8008` and follows the leader address
    automatically.
  - **Slot continuity strategies A/B/C** — Patroni
    `permanent_slots` (preferred) → PG-17+ logical-slot
    failover → fall-back recreate-and-resync.
  - **Dual-slot mode** — for tier-0 deployments, two
    independent slots (one on the leader, one on a chosen
    replica) provide survivable WAL streaming across
    multi-second failovers.
  - **Sync-target mode** — opt-in synchronous replication
    where the backup's WAL flush gates commit.

See the
[Patroni failover deep-dive](../explanation/patroni-failover-deep-dive.md)
for what each mechanism trades off.

### 4. LLM-assisted operations

The 3am operator gets a grounded co-pilot:

  - **`pg_hardstorage llm chat`** — read-only by default
    (advise+execute is opt-in).  Every claim the model
    makes is grounded in tool results from the live
    cluster — no model-trained-on-stale-docs surprises.
  - **Five safety gates** — preview-required before any
    suggested mutation, replay-protected execute, typed
    confirmation for destructive ops, RBAC + n-of-m, +
    anomaly refusal.
  - **Encrypted history** — every prompt + tool call +
    response stored AES-GCM-encrypted under a HKDF-derived
    DEK, scoped per principal.
  - **Signed evidence bundles** — `pg_hardstorage llm
    export-session` produces a cosign-signed bundle the
    auditor can verify offline.
  - **OpenAI-compatible wire format** — Anthropic, OpenAI,
    Ollama, vLLM, and any OpenAI-compatible vendor work
    through the same provider plugin.

The [LLM safety stack explanation](../explanation/llm-safety-stack.md)
walks through each gate.

### 5. Two verifier sandboxes

`pg_hardstorage verify --full` runs `pg_verifybackup`
inside an isolated environment:

  - **Docker** (default) — testcontainers + the official
    `postgres:<major>` image; the universal sandbox.
  - **Firecracker microVM** (`-tags firecracker`) — boots
    an operator-supplied vmlinux + rootfs.ext4, mounts the
    PGDATA image read-only, captures serial console.
    Strongest isolation posture available — no shared
    kernel with the host.

[Verify-sandbox tradeoffs](../explanation/verify-sandbox-tradeoffs.md)
documents when to pick which.

## Compliance and governance

The auditor's first questions all have ready answers in
the [Compliance section](../compliance/index.md):

  - **GDPR Article 17** — `pg_hardstorage kms shred`
    schedules KEK destruction; tenant-scoped backups
    become unrecoverable on the cloud's deletion cadence.
    Per-tenant isolation by default.
  - **Audit evidence bundles** — every audit event is
    hash-chained into a Merkle log; periodic anchoring to
    Rekor / a customer transparency log; export-bundle
    produces the offline-verifiable forensic package.
  - **SLSA L3 build provenance** — cosign-signed binaries
    + reproducible builds verified weekly via the new
    packaging CI workflow.
  - **Data residency pinning** — per-deployment region
    constraints; a backup bound to `eu-central-1` will
    refuse to replicate elsewhere.
  - **Framework matrices** — SOC 2 (CC, A, PI, C, P), ISO
    27001:2022 Annex A, HIPAA Technical Safeguards, PCI
    DSS v4.0 (req. 3 / 9 / 10), FedRAMP Moderate.
    Generated from
    `internal/compliance/controls.go`; the binary's
    behaviour is the source of truth.

Plus governance primitives:

  - **n-of-m approval** — destructive ops (`kms shred`,
    `repo gc --delete`, `backup delete --force`) require
    a configurable threshold of operator approvals.
  - **JIT access** — `pg_hardstorage jit issue` produces
    short-lived signed tokens for emergency operations.
  - **Threshold-attestation gate** — restores can be
    gated on a k-of-n threshold signature on the
    manifest.
  - **Legal hold** — `pg_hardstorage hold add/list/remove`
    suspends retention pruning per backup.
  - **Data-classification tags** — every backup carries a
    classification (`public` / `internal` / `confidential`
    / `restricted`) that drives which residency / retention
    rules apply.
  - **Cross-account replication ACL** — bilateral consent
    files (`source.json` + `accept.json`, both Ed25519-
    signed) gate every cross-account chunk-and-manifest
    copy.
  - **Insider-threat anomaly detection** —
    `pg_hardstorage insider scan` flags out-of-pattern
    egress + unusual destructive-op cadences.
  - **SCIM 2.0** — IdP-driven user / group provisioning
    primitive.
  - **i18n** — `en` / `de` / `fr` / `ja` translation
    catalogues for the CLI output.

## Operations

  - **Per-deployment SLOs as code** — `pg_hardstorage slo
    set --rpo 15m --rto 4m` declares the targets;
    `pg_hardstorage slo report` rolls up actuals;
    Prometheus alerts fire when actuals miss targets.
  - **Capacity planning** — `pg_hardstorage capacity
    report` projects 30 / 90 / 365-day repo growth from
    the past 30 days of throughput.
  - **Cost reporting** — per-deployment + per-tenant repo
    cost exports.
  - **Recovery drills** — `pg_hardstorage recovery drill`
    spins up a sandbox + restores + `pg_amcheck`'s the
    result; flagged as a verifiable event in the audit
    chain.
  - **Game days** — `pg_hardstorage gameday run` injects a
    chosen failure (storage 5xx, KEK unreachable, slot
    drop) and verifies the recovery posture.
  - **Repo audit + integrity checks** —
    `pg_hardstorage repo audit` + `integrity verify` walk
    the repo's chunk inventory and manifest signatures
    end-to-end.
  - **Repo bundle export / import** — air-gapped or
    one-way transfer via signed tarball bundles.
  - **Compliance reports** —
    `pg_hardstorage compliance report` walks the audit
    chain, encryption coverage, verification activity,
    approval / hold lifecycle and renders a verdict table
    per framework as Markdown / JSON / PDF.

## Plugin model

  - **Tier-1** — in-tree Go interfaces compiled into the
    main binary.  All five KMS providers, five storage
    backends, eleven renderers, fourteen sinks, eight LLM
    providers ship as Tier-1.
  - **Tier-2** — out-of-tree plugins discovered via
    `$HSPLUGIN_PATH` and the stdio JSON-RPC `--probe`
    handshake.  An operator can write a `mem://` storage
    plugin in Go, place the binary on `$HSPLUGIN_PATH`,
    and `pg_hardstorage doctor` will register the new
    `mem://` URL scheme.

The [build-a-storage-plugin tutorial](../tutorials/build-a-storage-plugin.md)
walks through a working Tier-2 plugin in 30 minutes.

## Build and packaging

  - **Single static binary** — `make build` produces
    `bin/pg_hardstorage`.  Default flavour is
    `CGO_ENABLED=0` for portability.
  - **FIPS variant** — `make build-fips` builds a
    Linux/amd64 binary against Go's BoringCrypto
    experiment.
  - **PKCS#11 variant** — `make build-pkcs11` activates
    the cgo-backed HSM provider via
    `github.com/miekg/pkcs11`.
  - **Firecracker variant** — `make build-firecracker`
    activates the microVM verifier-sandbox backend.
  - **Debian / RPM packages** — full `debian/` skeleton
    with `dpkg-buildpackage` + lintian-clean (per-PR L1
    gate) + piuparts + reprotest + fresh-VM smoke matrix
    (weekly L4 gate) + reproducible-build verification.
  - **`pg-hardstorage-cluster` wrapper** — Debian-friendly
    multi-tenant systemd template invocation, mirrors the
    `pg_ctlcluster` UX PostgreSQL admins expect.

## Upgrading

If you've been running `v0.9.x`, the v1.0 upgrade is
**configuration-compatible**.  No manifest rewrite, no
config migration, no behavioural break.

  - **Manifest schema** — unchanged from v0.9; v1.0
    commits the schema for 24 months.
  - **Output JSON** — `pg_hardstorage.v1` schema
    string is unchanged.
  - **CLI** — every command and flag from v0.9 still
    works; new flags are additive.
  - **REST API** — `/v1/...` endpoints are unchanged.

For step-by-step upgrade procedures, see
[upgrade procedures](../operations/upgrade-procedures.md).

## Known issues

The agents that wrote the v1.0 documentation grounded every
claim in source code and surfaced 15 SPEC ↔ code drift
items they couldn't fully reconcile.  All are tracked in
[`SPEC_DRIFT.md`](../SPEC_DRIFT.md):

  - 5 items where the code does less than SPEC promises
    (e.g. AES-256-GCM-SIV is undelivered; the code ships
    plain AES-256-GCM with random nonces).
  - 4 items where SPEC describes future-state behaviour
    that v1.0 doesn't ship (e.g. Rekor anchoring; metric
    registry).
  - 3 godoc clarifications.
  - 3 product calls needed.

None block v1.0 functionally.  The next engineering pass
(v28 audit) closes the gaps.
